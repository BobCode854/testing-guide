Process of evaluating software to ensure it meets requirements and works as expected.
Definition
  • 🎯 Executing software with intent to find defects before users do
  • 📋 Verifying software meets specified requirements
  • ✅ Validating software meets real user needs
  • 🔄 Can be performed manually or automatically
Why Testing Matters
  • 💰 Fixing bugs in production is 100x more expensive than fixing in dev
  • 🛡️ Prevents security vulnerabilities from reaching users
  • 📈 Builds confidence in the software quality
  • ⚙️ Critical phase of SDLC

Error → Defect → Failure: three distinct concepts on the defect chain.

Defect / BugThe flaw in the code artifactThe incorrect line of code in the source file
FailureObservable wrong behaviour at runtimeApp crashes, shows wrong total, or rejects valid input

The Chain
  • 1️⃣ Developer makes an Error (wrong logic)
  • 2️⃣ Error creates a Defect in the code
  • 3️⃣ When defect executes, it causes a Failure visible to users

A test case is a documented set of conditions and steps to verify a specific behaviour.
Components of a Test Case
  • 🆔 Test Case ID — unique identifier (e.g., TC_LOGIN_001)
  • 📝 Title / Description — what is being tested
  • 🔑 Pre-conditions — what must be true before test runs (user is on login page)
  • 👣 Test Steps — numbered, reproducible actions
  • 🗃️ Test Data — specific inputs (username, password values)
  • ✅ Expected Result — what SHOULD happen
  • 📊 Actual Result — what DID happen (filled during execution)
  • 🏷️ Pass / Fail Status + Priority + Requirement ID
Qualities of a Good Test Case
  • ⚛️ Atomic — tests one thing only
  • 🔁 Repeatable — same result every run
  • 🧩 Independent — doesn't depend on other test cases
  • 📖 Clear — anyone can execute it without asking questions

SDLC — a structured process for planning, creating, testing, and delivering software.
7 Phases
  • 1️⃣ Planning — define scope, feasibility analysis
  • 2️⃣ Requirements Analysis — gather & document requirements (SRS)
  • 3️⃣ System Design — architecture, DB schema, UI wireframes
  • 4️⃣ Implementation / Coding — developers write code
  • 5️⃣ Testing — QA finds and reports defects
  • 6️⃣ Deployment — release to production
  • 7️⃣ Maintenance — bug fixes, enhancements, new features
Common SDLC Models
  • 🌊 Waterfall — sequential, each phase completes before next
  • 🔁 Agile / Scrum — iterative sprints, continuous delivery
  • 🅥 V-Model — testing mirrored with each development phase
  • 🌀 Spiral — risk-driven iterative model

STLC — 6-phase sequence of testing activities that runs in parallel with SDLC.
6 Phases of STLC
  • 1️⃣ Requirements Analysis — QA reviews requirements for testability, creates RTM skeleton
  • 2️⃣ Test Planning — scope, approach, resources, timeline, test plan document
  • 3️⃣ Test Case Design — write test cases, prepare test data, scripts
  • 4️⃣ Test Environment Setup — configure hardware/software, set up tools
  • 5️⃣ Test Execution — run tests, log defects, retest fixed bugs
  • 6️⃣ Test Closure — analyze metrics, archive artifacts, lessons-learned report
  • 📌 STLC runs in parallel with SDLC phases, not after

Hard Assertion (Assert)
  • 🛑 Stops execution on first failure
  • ❌ Subsequent assertions are never run
  • ✅ Use when failure makes rest meaningless
Soft Assertion (SoftAssert)
  • ▶️ Collects ALL failures, continues
  • 📋 Reports all at assertAll()
  • ✅ Use for multiple independent checks
// Soft Assertion — all 3 run even if first fails SoftAssert sa = new SoftAssert(); sa.assertTrue(page.isNameDisplayed(), "name missing"); sa.assertTrue(page.isEmailDisplayed(), "email missing"); sa.assertTrue(page.isPhoneDisplayed(), "phone missing"); sa.assertAll(); // reports all failures here

Steady degradation under constant load is a resource leak signature — most commonly memory leak, connection leak, or unbounded accumulation.
What It Signals
  • 🧠 Memory leak: heap fills, GC thrashes, eventually OOM — visible as rising heap with shrinking reclaimed memory per cycle
  • 🔌 Unclosed DB connections exhausting the connection pool
  • 📦 Growing cache / collections without eviction policy
  • 📄 Ever-larger log files consuming disk and I/O
  • 📊 Table/queue growing without cleanup slowing queries over time
Investigation Steps
  • 📈 Correlate response-time curve with server metrics (heap, GC, connections, query time)
  • 🔍 Rising heap + shrinking GC reclaim → heap dump with Eclipse MAT
  • 🔗 Climbing connection count → connection leak in code
  • 🐢 Growing query time → unindexed accumulating table
  • ⏱️ This is precisely what endurance/soak testing is designed to catch
  • ✅ Fix retention (close resources, bound cache) then re-run soak to confirm flat profile
