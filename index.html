1. What Is Software Testing?

Foundations: definitions, history, the testing process, and core concepts every QA engineer must know.

Software Testing = systematically executing software to find defects, verify requirements, and ensure the product does what it should — and nothing it shouldn't.

Two Core Questions
  • ✅ "Does the software do what it is supposed to do?"
  • 🚫 "Does it avoid doing what it should not do?"
Evolution
  • 🏛️ Old: afterthought — final gate before release
  • 🔁 Today: continuous, integral part of SDLC
  • ⬅️ Shift-left: test early where fixes are cheapest
Key Distinctions
  • 🔍 Testing — finds defects; observes behavior, reports discrepancies; done by QA
  • 🔧 Debugging — fixes defects; locates root cause in code; done by developers
  • 📐 Verification — "Are we building the product right?" — reviews, inspections, static analysis
  • 🎯 Validation — "Are we building the right product?" — functional testing, UAT, real user needs

The Cost of Late Defect Detection

A production defect can cost 100x more than a requirements-phase fix — this is the core argument for shift-left testing.

Phase Defect FoundRelative Cost to Fix
Requirements1x
Design3x – 6x
Coding / Development10x
Testing / QA15x – 40x
Production / Release40x – 100x+
Key Insight
The earlier a defect is detected, the cheaper it is to fix. A requirements-level fix is a sentence edit; a production-level fix may involve a hotfix, regression testing, customer communication, and reputational cost.

Software Testing Life Cycle (STLC) — 6 Phases

1. Requirements Analysis

QA studies requirements to identify what is testable, raises clarifications, and defines testing scope and feasibility (manual vs automation).

2. Test Planning

The Test Lead defines strategy, effort estimation, resource planning, tools, schedule, and the entry/exit criteria for the testing effort.

3. Test Case Design

Detailed test cases and test scripts are written, along with test data preparation and requirement-to-test traceability mapping.

4. Test Environment Setup

The hardware, software, network, and test data needed to execute tests are configured. A smoke test confirms environment readiness.

5. Test Execution

Test cases are executed, actual vs expected results are compared, defects are logged and tracked, and re-testing/regression is performed.

6. Test Closure

Test completion criteria are evaluated, metrics and a test summary report are produced, and lessons learned are documented for the future.

The Testing Pyramid

Many fast unit tests at the base → fewer integration → few slow E2E at the top. Invert = ice cream cone anti-pattern.

/\ /E2E\ ← few, slow, expensive /------\ /Integr. \ ← some /----------\ / Unit Tests \ ← many, fast, cheap /--------------\

2. Types of Testing

Testing is grouped into Functional, Non-Functional, and Structural categories — each answering a different quality question.

Functional Testing

🔬 Unit Testing

Tests the smallest isolated unit of code (a method or class) in isolation from the rest of the system. Catches logic errors at the source.

When: Development phase
Example: Testing a login() method returns true for valid credentials.
🔗 Integration Testing

Tests that two or more modules work together correctly. Verifies the interfaces and data flow between components.

When: After unit testing
Example: Testing UserService + Database layer together.
🖥️ System Testing

Tests the complete, integrated system end-to-end against the requirements. Validates the full application behaves as a whole.

When: After integration
Example: Testing the entire checkout flow.
👥 UAT

User Acceptance Testing: business stakeholders validate the software against real-world business requirements before go-live.

When: Pre-release
Example: Client verifies invoice generation is correct.
💨 Smoke Testing

A quick, broad sanity check after a new build to confirm critical paths work. If smoke fails, the build is rejected.

When: Every build
Example: Can you log in and navigate the app?
🎯 Sanity Testing

A narrow, deep check of a specific area after a minor fix or change. Confirms the change works without full regression.

When: After a minor patch
Example: Verify only the bug-fixed feature behaves.
🔄 Regression Testing

Re-runs existing tests to ensure that new code changes have not broken previously working functionality.

When: After any change
Example: Full regression suite run after a sprint.
✅ Re-testing

Verifies that a specific, previously-failing defect has actually been fixed. Uses the same data and steps that caused the bug.

When: After dev fixes a defect
Example: Bug DEF-123 no longer reproduces.

Non-Functional Testing

⚡ Performance Testing

Measures responsiveness, throughput, and resource usage under a given workload. The umbrella for load and stress testing.

Example: API responds in under 200ms.
📈 Load Testing

Evaluates system behavior under expected, realistic user load to confirm it meets performance targets.

Example: 1,000 users hitting the app simultaneously.
💥 Stress Testing

Pushes the system beyond normal capacity to find its breaking point and how it recovers from failure.

Example: 10,000 concurrent users to find the breaking point.
🔒 Security Testing

Identifies vulnerabilities, threats, and risks, and verifies data protection and access controls.

Example: SQL injection, XSS attacks, broken auth.
🎨 Usability Testing

Assesses how intuitive and user-friendly the application is for real end users completing real tasks.

Example: Can new users complete checkout without help?
🌐 Compatibility Testing

Verifies the app works correctly across different browsers, operating systems, devices, and resolutions.

Example: Chrome / Firefox / Safari / IE11.

Structural (Box) Testing

⬜ White Box Testing

Tester has full knowledge of the internal code and structure. Focuses on logic paths and code coverage.

Techniques: Statement coverage, branch coverage, path coverage.
⬛ Black Box Testing

Tester has no knowledge of internal code; tests only against inputs and expected outputs.

Techniques: BVA, Equivalence Partitioning, Decision Tables.
🔳 Grey Box Testing

Tester has partial knowledge of internals — combines black box behavior with structural insight.

Best for: Integration testing, security testing.

Black Box Technique — Equivalence Partitioning & BVA

// Black Box: Equivalence Partitioning for age validation // Valid partition: 18-65 | Invalid: <18 | Invalid: >65 @Test void validateAge_ValidRange_ReturnsTrue() { assertTrue(validator.isValidAge(30)); } @Test void validateAge_BelowMin_ReturnsFalse() { assertFalse(validator.isValidAge(15)); } @Test void validateAge_AboveMax_ReturnsFalse() { assertFalse(validator.isValidAge(70)); } // Boundary Value Analysis: test 17, 18, 19, 64, 65, 66 @Test void validateAge_LowerBoundary_ReturnsFalse() { assertFalse(validator.isValidAge(17)); } @Test void validateAge_LowerBoundary_ReturnsTrue() { assertTrue(validator.isValidAge(18)); }

3. Automation Testing

When and how to automate, framework architectures, ROI, and CI/CD integration.

What Is Test Automation?

Test automation uses scripts/tools to execute tests, compare results, and report outcomes — without human intervention, at machine speed.

Why Automate
  • ⚡ Regression suite runs in minutes vs days
  • 🔁 Repeatable on every commit — zero human effort
  • 📊 Data-driven: same logic, hundreds of inputs
  • 🌐 Cross-browser matrix would multiply manual effort
When NOT to Automate
  • 🧭 Exploratory testing — needs human curiosity
  • 🎨 UX/usability — subjective human judgment
  • 🚧 Rapidly-changing UI — maintenance cost exceeds benefit
  • 1️⃣ One-time tests — scripting cost > single-run value

When to Automate — Decision Checklist

Test TypeAutomate?Reason
Repetitive regression tests✅ YesRun on every build; huge cumulative time savings
Data-driven tests (many inputs)✅ YesSame logic, many datasets — ideal for parameterization
Smoke / sanity tests✅ YesFast, frequent, gate every deployment
Performance / load tests✅ YesImpossible to simulate thousands of users manually
Cross-browser / cross-platform✅ YesMatrix of environments multiplies manual effort
Stable, mature features✅ YesLow maintenance — UI/logic rarely changes
One-time / throwaway tests❌ NoScripting cost exceeds the benefit of a single run
Exploratory testing❌ NoRequires human intuition and curiosity
Usability / UX testing❌ NoSubjective human judgment cannot be scripted
Rapidly changing / unstable UI❌ NoScripts break constantly; maintenance cost too high

Automation ROI Formula

Break-Even Calculation

Break-even runs = Automation build cost ÷ (Manual execution time per run)

  • 📌 Example: 30 min manual × 100 runs/year = 50 hrs saved/year
  • 🔨 If automation costs 10 hrs to build → break-even at ~20 runs
  • 💰 Year-one net saving = 40 hrs. Savings grow every year thereafter

CI/CD Integration

Automate the pipeline — tests run on every push, providing fast feedback and gating deployments.

  • 1️⃣ Commit — developer pushes code to Git
  • 2️⃣ Trigger — CI server (Jenkins / GitHub Actions / GitLab CI) starts pipeline
  • 3️⃣ Build — compile & package (mvn clean install); fail fast on compile errors
  • 4️⃣ Unit Tests — fast tests run first; failure fails the build immediately
  • 5️⃣ Deploy to Test Env — artifact pushed to staging
  • 6️⃣ Automated Suite — Selenium/TestNG regression + smoke (parallel via Grid)
  • 7️⃣ Report — Allure / Extent / Surefire results & screenshots published as artifacts
  • 8️⃣ Gate / Promote — all pass → promote build; failures → notify team immediately

4. TOSCA Deep Dive

Tricentis TOSCA — model-based, scriptless, continuous testing for the enterprise.

TOSCA = commercial, model-based, scriptless enterprise automation — scan app → create Modules → assemble tests, no coding needed.

Key Strengths
  • 🧩 Scriptless — business analysts can build automation
  • 🏗️ Model-based — one Module update fixes all dependent tests
  • 🌐 Unified: Web + Mobile + Desktop + API from one tool
  • 🏢 Deep SAP & Salesforce certified support
  • 🤖 AI self-healing — adapts when UI changes
Key Facts
  • 💰 Commercial license — expensive
  • 🏭 Widely used in regulated industries (banking, pharma)
  • 🔒 Vendor lock-in to Tricentis ecosystem
  • 🧠 Low coding barrier — but requires TOSCA expertise

Philosophy & Architecture

Core separation: Modules (app structure) × TestCases (logic) × TestData (values) — change one layer without touching the others.

TOSCA Workspace/ ├── Requirements/ ├── TestCases/ │ ├── Regression/ │ ├── Smoke/ │ └── UAT/ ├── Modules/ │ ├── CommonModules/ │ └── ApplicationPages/ ├── TestData/ │ └── DataPools/ ├── ExecutionLists/ │ ├── Nightly/ │ └── CI_Pipeline/ └── Reports/

Key Components

ComponentDescription
ModulesAbstraction of UI / API components, auto-created by the ScanEngine. Reused across many test cases.
TestCasesExecutable tests built from ordered TestSteps that reference Modules.
TestStepBlocksReusable logic blocks (e.g., "Login") shared across multiple test cases.
Business ParametersCross-case parameter binding so values flow between test cases.
ExecutionListsOrdered groups of test cases assembled specifically for execution runs.
TDM (Test Data Management)Parameterized data pools and synthetic test data generation.
Risk Coverage OptimizerMaps test cases to business risks and prioritizes tests by risk weight.

How the ScanEngine Works

  • 1️⃣ Point ScanEngine at a running screen, dialog, or API endpoint
  • 2️⃣ TOSCA inspects HTML DOM, controls, and fields
  • 3️⃣ Tester selects which elements to steer or verify
  • 4️⃣ TOSCA generates a Module with stable identification properties
  • 5️⃣ Module becomes a reusable building block referenced by any TestCase
  • 6️⃣ UI changes? Re-scan single Module → all tests inherit the fix automatically

Storyboard, API Testing, DEX & AI

Storyboard Testing
  • 🖼️ Design tests visually using screenshots
  • 🖱️ Drag & annotate elements directly on screen image
  • 📄 Serves as living documentation of the user journey
API Testing
  • 🌐 Supports REST, SOAP, and GraphQL
  • 🔑 Auth: Basic, OAuth, tokens — all configurable
  • 🔗 Mix UI + API steps in same TestCase (true E2E)
DEX (Distributed Execution)
  • 🤝 Central DEX server coordinates multiple AOS Agents
  • ⚡ Runs test cases in parallel across machines
  • 🌍 Concurrent cross-browser & cross-environment execution
AI Self-Healing
  • 🤖 Detects when element ID/properties change
  • 👁️ Uses Vision AI for visual re-identification
  • 🩹 Keeps suites green without manual locator fixes

Annotation Execution Order

TestNG lifecycle flows top-to-bottom on entry, bottom-to-top on exit — wrapping each test in predictable setup/teardown layers.

@BeforeSuite → @BeforeTest → @BeforeClass → @BeforeGroups → @BeforeMethod @Test ← the actual test executes here @AfterMethod → @AfterGroups → @AfterClass → @AfterTest → @AfterSuite
@BeforeSuite public void suiteSetup() { System.out.println("1. Before Suite"); } @BeforeTest public void testSetup() { System.out.println("2. Before Test"); } @BeforeClass public void classSetup() { System.out.println("3. Before Class"); } @BeforeMethod public void methodSetup() { System.out.println("4. Before Method"); } @Test public void testCaseOne() { System.out.println("5. The Test"); } @AfterMethod public void methodTeardown(){ System.out.println("6. After Method"); } @AfterClass public void classTeardown() { System.out.println("7. After Class"); } @AfterTest public void testTeardown() { System.out.println("8. After Test"); } @AfterSuite public void suiteTeardown() { System.out.println("9. After Suite"); }

@DataProvider — Data-Driven Testing

@DataProvider returns a 2D Object[][] — TestNG runs the test once per row. Set parallel=true to run all rows concurrently.

@DataProvider(name = "loginData", parallel = true) public Object[][] provideLoginData() { return new Object[][] { {"validUser", "validPass123", true}, {"invalidUser", "wrongPass", false}, {"", "", false}, {"validUser", "", false} }; } @Test(dataProvider = "loginData") public void testLogin(String username, String password, boolean expectedResult) { boolean result = loginService.authenticate(username, password); Assert.assertEquals(result, expectedResult, "Login result mismatch for user: " + username); }

Listeners

Listeners implement ITestListener to hook test lifecycle events — capture screenshots on failure, enrich reports, and add custom logging without modifying test code.

public class TestListener implements ITestListener { @Override public void onTestFailure(ITestResult result) { System.out.println("FAILED: " + result.getName()); // Take screenshot TakesScreenshot ts = (TakesScreenshot) DriverManager.getDriver(); File src = ts.getScreenshotAs(OutputType.FILE); try { FileUtils.copyFile(src, new File("screenshots/" + result.getName() + ".png")); } catch (IOException e) { e.printStackTrace(); } } @Override public void onTestSuccess(ITestResult result) { System.out.println("PASSED: " + result.getName()); } }

Soft Assertions

SoftAssert collects ALL failures and reports them together at assertAll() — validate multiple conditions in one test without stopping at the first failure.

@Test public void verifyProfilePage() { SoftAssert softAssert = new SoftAssert(); softAssert.assertEquals(profile.getName(), "John Doe", "Name mismatch"); softAssert.assertEquals(profile.getEmail(), "john@test.com", "Email mismatch"); softAssert.assertTrue(profile.isActive(), "User should be active"); // All collected failures reported here: softAssert.assertAll(); }

IRetryAnalyzer — Auto-Retry Flaky Tests

Implement IRetryAnalyzer to automatically re-run a failing test up to N times before marking it failed — use sparingly, fix root causes instead.

public class RetryAnalyzer implements IRetryAnalyzer { private int count = 0; private static final int MAX_RETRY = 2; @Override public boolean retry(ITestResult result) { if (count < MAX_RETRY) { count++; System.out.println("Retrying test: " + result.getName() + " attempt " + count); return true; } return false; } }

String mainWindow = driver.getWindowHandle(); Set<String> allWindows = driver.getWindowHandles(); for (String window : allWindows) { if (!window.equals(mainWindow)) { driver.switchTo().window(window); // perform actions driver.close(); } } driver.switchTo().window(mainWindow);

Page Object Model (POM)

POM = one class per page, encapsulating locators + actions — tests stay clean, and UI changes are one-place fixes.

  • 📦 Each page → its own Java class with @FindBy annotated fields
  • 🏭 PageFactory.initElements() lazily initializes elements on first use
  • 🔗 Actions return the next page object — enables fluent test chaining
  • 🧹 Tests contain no locators — only readable method calls
// BasePage.java public abstract class BasePage { protected WebDriver driver; protected WebDriverWait wait; public BasePage(WebDriver driver) { this.driver = driver; this.wait = new WebDriverWait(driver, Duration.ofSeconds(10)); PageFactory.initElements(driver, this); } protected WebElement waitForClickable(WebElement element) { return wait.until(ExpectedConditions.elementToBeClickable(element)); } protected void type(WebElement element, String text) { waitForClickable(element).clear(); element.sendKeys(text); } protected void click(WebElement element) { waitForClickable(element).click(); } } // LoginPage.java public class LoginPage extends BasePage { @FindBy(id = "username") private WebElement usernameField; @FindBy(id = "password") private WebElement passwordField; @FindBy(css = "button[type='submit']") private WebElement loginButton; @FindBy(className = "error-message") private WebElement errorMessage; public LoginPage(WebDriver driver) { super(driver); } public DashboardPage loginAs(String username, String password) { type(usernameField, username); type(passwordField, password); click(loginButton); return new DashboardPage(driver); } public boolean isErrorDisplayed() { return errorMessage.isDisplayed(); } } // LoginTest.java public class LoginTest extends BaseTest { @Test public void validLogin_RedirectsToDashboard() { DashboardPage dashboard = new LoginPage(driver) .loginAs("admin", "admin123"); assertTrue(dashboard.isWelcomeMessageDisplayed()); } }

Actions Class (Keyboard & Mouse)

The Actions class chains complex interactions — hover, right-click, drag-and-drop, keyboard combos — all executed with .perform().

Actions actions = new Actions(driver); // Hover over a menu actions.moveToElement(menu).perform(); // Right-click (context menu) actions.contextClick(element).perform(); // Double-click actions.doubleClick(element).perform(); // Drag and drop actions.dragAndDrop(source, target).perform(); // Keyboard combo (Ctrl + A) actions.keyDown(Keys.CONTROL).sendKeys("a").keyUp(Keys.CONTROL).perform();

JavascriptExecutor

Use JavascriptExecutor when Selenium's native API can't interact — click via JS, scroll, set values, read DOM properties.

JavascriptExecutor js = (JavascriptExecutor) driver; // Click via JS (when normal click fails) js.executeScript("arguments[0].click();", element); // Scroll element into view js.executeScript("arguments[0].scrollIntoView(true);", element); // Scroll to bottom of page js.executeScript("window.scrollTo(0, document.body.scrollHeight);"); // Set value directly js.executeScript("arguments[0].value='hello';", inputField); // Get page title String title = (String) js.executeScript("return document.title;");

Screenshot Utility

Capture screenshots on test failure using TakesScreenshot — best called from a TestNG @AfterMethod or Listener's onTestFailure().

public static String captureScreenshot(WebDriver driver, String name) throws IOException { TakesScreenshot ts = (TakesScreenshot) driver; File src = ts.getScreenshotAs(OutputType.FILE); String path = "screenshots/" + name + "_" + System.currentTimeMillis() + ".png"; FileUtils.copyFile(src, new File(path)); return path; }

Selenium Grid & RemoteWebDriver

Grid distributes tests across machines/browsers in parallel — connect via RemoteWebDriver, hub routes to the matching node.

  • 🔀 Hub-node (Selenium 3) or unified/distributed (Selenium 4)
  • 📡 Tests point to http://hub:4444/wd/hub — nothing else changes
  • 🐳 Docker Compose with Selenium Grid 4 containers = easiest setup
  • ⚡ Combine with TestNG parallel="methods" for maximum speed
// Connect to a remote Grid hub ChromeOptions options = new ChromeOptions(); options.setBrowserVersion("latest"); options.setPlatformName("Linux"); WebDriver driver = new RemoteWebDriver( new URL("http://localhost:4444/wd/hub"), options ); driver.get("https://example.com"); System.out.println(driver.getTitle()); driver.quit();
